## 1. Introduction

### 1.1 Purpose

This document defines the complete software requirements for a digital PDF bookstore platform. The system allows a single seller (admin) to list and sell PDF-format books, and buyers to browse, purchase, and download them securely.

### 1.2 Scope

The system consists of three independent applications sharing one backend API:

- **Backend** — RESTful API built with Node.js and Express, connected to MongoDB
- **Buyer App** — React SPA for customers to browse, purchase, and download books
- **Admin App** — React SPA for the store owner to manage books, orders, and customers

### 1.3 Intended Audience

- Development team building the application
- Project stakeholders reviewing requirements
- QA team planning test coverage

### 1.4 Definitions

| Term | Meaning |
|---|---|
| Buyer | A registered customer who browses and purchases books |
| Admin | The single store owner who manages the entire platform |
| Digital Delivery | Secure, time-limited PDF download after payment |
| Presigned URL | A time-limited signed URL generated by the backend to allow one-time file access from Cloudflare R2 |
| RTK Query | Redux Toolkit's built-in data fetching and caching layer |
| R2 | Cloudflare R2 — S3-compatible object storage with zero egress fees |

## 2. Overall Description

### 2.1 Product Perspective

A standalone web-based e-commerce platform specialising in digital books in PDF format. No physical shipping is involved. All file delivery is handled via secure signed URLs from Cloudflare R2.

### 2.2 Product Functions (Summary)

**Buyer App:**
- Register, login (email/password or Google One Tap)
- Browse, search, and filter books
- Add to cart and wishlist
- Checkout and pay via Razorpay
- Download purchased PDFs from "My Library"
- Leave ratings and reviews

**Admin App:**
- Manage book catalogue (add, edit, delete, publish/unpublish)
- Upload PDF files and cover images to Cloudflare R2
- View and manage orders
- View and manage customer accounts
- Configure store settings

### 2.3 User Classes

| User Class | Description |
|---|---|
| Guest | Unauthenticated visitor — can browse and search but cannot purchase |
| Buyer | Registered customer — can purchase and download books |
| Admin | Store owner — full access to the admin application |

### 2.4 Operating Environment

- Modern web browsers (Chrome, Firefox, Safari, Edge)
- Node.js v24.x (Active LTS) on the server
- MongoDB 8.2.x (cloud hosted, e.g. MongoDB Atlas)
- Cloudflare R2 for file storage
- Razorpay for payment processing

## 4. Functional Requirements — Buyer App

### 4.1 Authentication & Profile

#### FR-B-01: User Registration
- A guest can register with full name, email, and password
- Password must be minimum 8 characters
- Email must be unique — duplicate email returns a 409 error
- Password is hashed using Argon2id before storage
- A verification email is sent on successful registration

#### FR-B-02: Email/Password Login
- A registered user can log in with email and password
- On success, the backend issues a JWT access token (15-minute expiry) and a refresh token stored in an `httpOnly` cookie (7-day expiry)
- Failed login returns a generic "Invalid credentials" message (no user enumeration)

#### FR-B-03: Google One Tap Login
- The login page displays a Google One Tap prompt automatically
- On user consent, the Google ID token is sent to `POST /api/auth/google`
- The backend verifies the token using `google-auth-library`, then finds or creates the user in MongoDB
- The same JWT access/refresh token pair is issued as for email login

#### FR-B-04: Token Refresh
- When the access token expires, the frontend silently calls `POST /api/auth/refresh` using the `httpOnly` refresh token cookie
- A new access token is returned without requiring re-login

#### FR-B-05: Logout
- Calling `POST /api/auth/logout` clears the refresh token cookie and invalidates the stored refresh token in the database

#### FR-B-06: Profile Management
- A logged-in buyer can update their display name and avatar
- A buyer can change their password (requires current password confirmation)
- A buyer can view their full order history

---

### 4.2 Book Discovery

#### FR-B-07: Home Page
- Displays three sections: Featured Books, New Releases, Bestsellers
- Each section shows a maximum of 8 books in a card grid

#### FR-B-08: Category Browsing
- Books are organised into categories (e.g. Fiction, Technology, Business, Science)
- A buyer can browse all books within a selected category

#### FR-B-09: Search
- A search bar is available on all pages
- Search queries against book title, author name, and description
- Results support filters: price range, category, minimum rating
- Results are paginated (12 books per page)

#### FR-B-10: Book Detail Page
- Displays: cover image, title, author, description, price, category, average rating, review count
- Shows a "Sample Preview" section if a sample PDF chapter exists
- Displays related books in the same category
- Shows "Already in your library" badge if the buyer has purchased the book

---

### 4.3 Purchase Flow

#### FR-B-11: Shopping Cart
- A buyer can add a book to the cart from the detail page or listing
- A buyer can remove a book from the cart
- Cart is persisted in Redux state (client-side only, not in the database)
- Duplicate items are not allowed — adding an owned book to the cart is blocked

#### FR-B-12: Wishlist
- A buyer can save books to a wishlist
- Wishlist is persisted in the database and synced on login
- A buyer can move items from the wishlist to the cart

#### FR-B-13: Checkout
- The checkout page displays cart items with total price
- On clicking "Pay Now", the frontend calls `POST /api/orders/create` to create a Razorpay Order on the backend
- The Razorpay checkout widget opens in the browser
- See Section 14 for the complete payment flow

#### FR-B-14: Order Confirmation Email
- After a verified successful payment, the system sends an order confirmation email to the buyer's registered email address
- The email includes: order ID, list of purchased books, total amount, and a link to "My Library"

---

### 4.4 Digital Delivery

#### FR-B-15: My Library
- A "My Library" page lists all books the buyer has successfully purchased
- Each entry shows: cover image, title, author, and a "Download PDF" button

#### FR-B-16: Secure PDF Download
- Clicking "Download PDF" calls `GET /api/library/:bookId/download`
- The backend verifies the buyer owns the book (checks order records)
- The backend generates a presigned `GetObject` URL from Cloudflare R2 with a 15-minute expiry
- The signed URL is returned to the frontend and opened in a new tab
- Download attempt count is tracked — a maximum of **5 downloads** per purchase is enforced

#### FR-B-17: Download Limit Enforcement
- If a buyer exceeds the 5-download limit, the API returns a 403 error with message "Download limit reached. Contact support."

---

### 4.5 Reviews & Ratings

#### FR-B-18: Submit Review
- A buyer can submit a star rating (1–5) and optional review text for a book they have purchased
- Only one review per buyer per book is allowed
- Submitting again replaces the existing review

#### FR-B-19: View Reviews
- The book detail page shows the average rating and all approved reviews
- Reviews are sorted by newest first
- Unverified or flagged reviews are hidden from public view

#### FR-B-20: Flag Review
- Any buyer can flag a review as inappropriate
- Flagged reviews are queued for admin moderation

---

### 4.6 Support

#### FR-B-21: Contact Form
- A contact form accepts: name, email, subject, and message
- On submission, the message is sent to the store's email via Nodemailer

#### FR-B-22: FAQ Page
- A static FAQ page is available with common questions and answers

## 5. Functional Requirements — Admin App

### 5.1 Dashboard

#### FR-A-01: Sales Overview
- Displays total revenue (all time), today's revenue, and this month's revenue
- Displays total number of orders (all time), today's orders
- Displays top 5 bestselling books by revenue

#### FR-A-02: Recent Orders Feed
- Shows the last 10 orders with: buyer name, book(s), amount, status, and timestamp

---

### 5.2 Book Management

#### FR-A-03: Add Book
- Admin can add a new book with: title, author, description, category, price (INR), cover image, PDF file, and optional sample chapter PDF
- Cover image is uploaded to the **public** Cloudflare R2 bucket
- PDF file and sample chapter are uploaded to the **private** Cloudflare R2 bucket
- The book is saved to MongoDB with `isPublished: false` by default

#### FR-A-04: Edit Book
- Admin can update any field of an existing book
- Re-uploading the PDF replaces the old file in R2 (old file is deleted)

#### FR-A-05: Delete Book
- Admin can delete a book
- Deleting a published book that has been purchased is **soft-deleted** (marked as `isDeleted: true`) — existing buyers retain download access
- Deleting an unpublished book with no orders is a **hard delete** (removed from database and R2)

#### FR-A-06: Publish / Unpublish
- Admin can toggle the published state of any book
- Unpublished books are not visible in the buyer app
- A published book with existing orders cannot be unpublished without a confirmation prompt

#### FR-A-07: Bulk Actions
- Admin can select multiple books and apply: Publish, Unpublish, or Delete in bulk

---

### 5.3 Order Management

#### FR-A-08: Order List
- Displays all orders with filters: status (pending, paid, failed, refunded), date range, and search by buyer email
- Paginated at 20 orders per page

#### FR-A-09: Order Detail
- Admin can view full order details: buyer info, books purchased, amount, payment ID, timestamp, and download count per book

#### FR-A-10: Manual Refund
- Admin can trigger a refund for a paid order via the Razorpay Refunds API
- On successful refund, the order status is updated to `refunded` and the buyer loses download access

---

### 5.4 Customer Management

#### FR-A-11: Customer List
- Displays all registered buyers with: name, email, registration date, total orders, and total spend
- Searchable by name or email

#### FR-A-12: Customer Detail
- Admin can view a specific buyer's full profile and complete purchase history

#### FR-A-13: Suspend / Ban Account
- Admin can suspend a buyer account, preventing login
- A suspended buyer attempting to login receives "Your account has been suspended. Contact support."

---

### 5.5 Content Moderation

#### FR-A-14: Review Moderation
- Admin can view all flagged reviews
- Admin can approve (make visible) or remove a flagged review

---

### 5.6 Settings

#### FR-A-15: Store Settings
- Admin can update: store name, store logo, contact email, and default currency (INR)

#### FR-A-16: Email Template Configuration
- Admin can edit the body text of the order confirmation email template

#### FR-A-17: Razorpay Webhook Configuration
- The settings page displays the correct Razorpay webhook endpoint URL and expected events for the admin to configure in the Razorpay dashboard

## 6. Non-Functional Requirements

### 6.1 Performance

| Requirement | Target |
|---|---|
| API response time (P95) | < 300ms for standard queries |
| Home page load time | < 2 seconds on 4G connection |
| PDF presigned URL generation | < 200ms |
| Search results | < 500ms |

### 6.2 Scalability

- All database queries on frequently filtered fields (category, isPublished, price) must have MongoDB indexes
- RTK Query caching on the frontend reduces redundant API calls
- File serving is handled entirely by Cloudflare CDN (zero origin egress cost)

### 6.3 Reliability

- The backend must handle Razorpay webhook retries idempotently (duplicate `payment.captured` events must not create duplicate orders)
- PDF download requests must not fail if R2 has a brief outage — retry logic should be implemented
- Database connections use Mongoose connection pooling

### 6.4 Maintainability

- All environment-specific configuration via `.env` files
- TypeScript is used across backend and both frontend apps
- ESLint + Prettier enforced across all packages
- Code coverage target: minimum 70% for backend routes, 60% for frontend components

### 6.5 Accessibility

- All buyer app pages must meet WCAG 2.1 AA standards
- DaisyUI components are built on semantic HTML — use the correct component variants
- All images must have descriptive `alt` attributes
- Keyboard navigation must work for all interactive elements

### 6.6 Browser Support

- Chrome (latest 2 versions)
- Firefox (latest 2 versions)
- Safari 16+
- Edge (latest 2 versions)
- Mobile: iOS Safari, Android Chrome

## 12. Security Requirements

### 12.1 Authentication

- JWT access tokens: 15-minute expiry, signed with HS256 (single service — no inter-service token verification needed)
- Refresh tokens: 7-day expiry, stored as Argon2id hash in MongoDB, sent only via `httpOnly`, `Secure`, `SameSite=Strict` cookie
- `algorithms` must be explicitly set in `jwt.verify()` to prevent algorithm confusion attacks (`"none"` algorithm)
- Refresh token rotation: each `/auth/refresh` call issues a new refresh token and invalidates the old one
- Google One Tap tokens verified server-side using `google-auth-library` — never trusted client-side

### 12.2 Password Security

- Argon2id hashing with OWASP 2026 recommended parameters (64 MB memory cost, 3 iterations)
- Minimum password length: 8 characters
- Login error messages are generic — no distinction between "email not found" and "wrong password"

### 12.3 Payment Security

- Razorpay webhook endpoint verifies the HMAC-SHA256 signature before processing any event
- Idempotency: duplicate `payment.captured` events with the same `razorpay_order_id` are silently ignored
- PDF download access is only granted after server-side signature verification — never after client-side success callback alone

### 12.4 File Security

- PDF files are stored in a **private** Cloudflare R2 bucket — no public URL is ever generated for PDF files
- Presigned download URLs expire in 15 minutes
- Cover images and sample previews are stored in a **public** bucket served via Cloudflare CDN
- File uploads validate content type and size server-side (PDF max: 50 MB, cover image max: 5 MB)

### 12.5 API Security

- `helmet` sets secure HTTP headers on all responses
- `cors` restricts allowed origins to the buyer and admin app domains only
- `express-rate-limit` on auth endpoints: 10 requests per minute per IP
- All request bodies are validated with Zod schemas before reaching controllers
- MongoDB queries use Mongoose — no raw string interpolation that could enable NoSQL injection

## 13. File Storage Strategy

### 13.1 Cloudflare R2 Bucket Structure

**Private Bucket** (`books-private`) — PDFs only, no public access:
```
pdfs/
  {bookId}-{uuid}.pdf         # Full book PDF
  samples/
    {bookId}-sample-{uuid}.pdf  # Sample chapter PDF
```

**Public Bucket** (`books-public`) — covers served via CDN:
```
covers/
  {bookId}-{uuid}.jpg         # Book cover image (optimised by Sharp)
```

### 13.2 Upload Flow (Admin)

1. Admin selects files in the Admin App file uploader
2. Frontend sends files to `POST /api/admin/books` as `multipart/form-data`
3. Backend receives files via Multer into memory buffer
4. Sharp resizes cover image to 400×600px WebP
5. `@aws-sdk/client-s3` `PutObjectCommand` uploads both files to R2
6. R2 keys are stored in the Book MongoDB document
7. Original files are never written to disk on the server

### 13.3 Download Flow (Buyer)

1. Buyer clicks "Download PDF" in My Library
2. Frontend calls `GET /api/library/:bookId/download`
3. Backend verifies ownership (Order record with `status: 'paid'`)
4. Backend checks download count ≤ 5
5. Backend generates presigned `GetObjectCommand` URL (15-minute expiry) from R2
6. Backend increments `downloadCount` in the Order document
7. Frontend receives the presigned URL and opens it in a new tab

## 14. Payment Flow

### 14.1 Razorpay Integration Flow

```
Buyer Frontend          Backend API              Razorpay
     │                       │                       │
     │── POST /orders/create─►│                       │
     │   { bookIds }          │── Create Order ──────►│
     │                        │◄── { order_id, amount }│
     │◄── { razorpayOrderId, ─│                       │
     │      amount, key_id }  │                       │
     │                        │                       │
     │── Open Razorpay Widget─►────────────────────────
     │         (Buyer enters card details)             │
     │◄── Payment success: ───────────────────────────│
     │   { payment_id,        │                       │
     │     order_id,          │                       │
     │     signature }        │                       │
     │                        │                       │
     │── POST /orders/verify─►│                       │
     │   { payment_id,        │                       │
     │     order_id,          │── Webhook: payment ──►│
     │     signature }        │   .captured event     │
     │                        │                       │
     │                        │ Verify HMAC signature │
     │                        │ Update Order status   │
     │                        │ Send confirmation email
     │◄── { success: true } ──│                       │
     │                        │                       │
     │── Redirect to My Library                        │
```

### 14.2 Webhook Handling

- Razorpay sends `payment.captured`, `payment.failed`, and `refund.processed` events to `POST /api/orders/webhook`
- The webhook endpoint is **not** behind JWT auth — it uses Razorpay HMAC signature verification instead
- The `razorpay_order_id` is used as the idempotency key — if an order is already `paid`, the webhook is acknowledged but no DB write occurs

## 15. Authentication Flow

### 15.1 Email/Password Login Flow

```
1. POST /api/auth/login { email, password }
2. Backend: find user by email
3. Backend: verify password with argon2.verify()
4. Backend: sign accessToken (JWT, 15min, HS256)
5. Backend: sign refreshToken (JWT, 7days, HS256)
6. Backend: hash refreshToken and store in User.refreshToken
7. Response: { accessToken } + Set-Cookie: refreshToken (httpOnly)
8. Frontend: store accessToken in Redux authSlice (memory only)
9. Frontend: axios interceptor adds Bearer token to all API requests
10. On 401: frontend calls POST /api/auth/refresh automatically
```

### 15.2 Google One Tap Flow

```
1. @react-oauth/google renders One Tap prompt on Login page
2. User clicks "Continue as [Name]"
3. Google returns credentialResponse.credential (ID token JWT)
4. Frontend: POST /api/auth/google { credential }
5. Backend: google-auth-library verifyIdToken()
6. Backend: extract { sub, email, name, picture } from payload
7. Backend: findOne({ googleId: sub }) OR findOne({ email })
8. If new user: create User document (no passwordHash)
9. If existing email user: link googleId to their account
10. Issue same accessToken + refreshToken as email login
11. Same response and cookie flow as email login
```

## 16. Constraints & Assumptions

### 16.1 Constraints

- Single seller model — there is exactly one admin account. Multi-seller support is out of scope.
- Currency is fixed to INR (Indian Rupees) as Razorpay is the payment gateway
- PDF files only — no ePub, audiobook, or other formats in scope
- Maximum book file size: 50 MB per PDF
- Maximum cover image size: 5 MB

### 16.2 Assumptions

- The admin account is seeded directly in the database — no admin registration UI is provided
- Cloudflare R2 is pre-configured by the developer (bucket creation, API token generation, CORS policy for public bucket)
- Razorpay account and webhook configuration are set up before deployment
- The application is deployed with HTTPS — `Secure` cookies will not work over HTTP
- MongoDB Atlas is used for hosting — no self-hosted MongoDB setup is covered

### 16.3 Out of Scope

- Subscription or recurring billing model
- Multi-currency support
- Coupon / discount codes
- Analytics dashboard (beyond basic sales overview)
- Mobile native app (iOS / Android)
- Book recommendation engine or AI features
- Multi-language / internationalisation (i18n)
- Affiliate or referral system
